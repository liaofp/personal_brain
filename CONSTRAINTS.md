# 代码生成专用指令

严格依照架构、层级功能、流转规则、技术规范，产出完整可运行Python工程代码。

适配环境：Windows10 + Python + SQLAlchemy + SQLite
架构层级：人机交互层→Agent智能代理层→Service业务服务层→DAO数据访问层→数据库基础模型层
运行模式：同步业务调用、异步事件联动双机制并行

## 一、固定目录结构

严格按如下结构创建文件，路径、名称不可改动：

```
personal_brain/
├── main.py

├── AGENTS.md
├── interaction/
│   └── interaction_layer.py
```

## 二、业务流转规范

### 2.1 同步业务链路
用户控制台输入 → 大模型接口做意图识别参数提取 → 路由分发指派对应Agent → Agent调用Service → Service调用DAO → DAO操作底层数据库模型

### 2.2 异步事件链路
模块状态/数据发生变更 → 向事件总线发布事件 → 订阅模块监听捕获事件 → 触发自身内部业务逻辑执行

### 2.3 常规增删改查走同步调用，跨模块联动、状态同步走事件总线

## 三、各层级必填实现功能

### 3.1 人机交互层 interaction_layer.py

1. 搭建控制台循环交互界面，接收用户自然语言输入
2. 对接大模型接口，完成语义理解、行为意图判定
3. 从对话文本中自动抽取姓名、时间、任务内容、联系方式等业务参数
4. 编写路由分发函数，根据识别结果匹配业务类型，下发至人员Agent或任务Agent
5. 接收底层处理结果，格式化中文反馈展示在控制台
6. 配置退出指令，输入指定关键词正常关闭程序
7. **不编写业务运算、数据存取逻辑**

### 3.2 Agent智能代理层

**person_agent.py 人员代理**
1. 接收交互层路由下发的人员管理任务与参数
2. 调用人员业务Service执行对应操作
3. 人员信息新增、修改、删除后，主动向事件总线推送状态变更事件
4. 订阅关联业务事件，监听到消息后触发对应联动处理
5. 汇总业务执行结果，回传给人机交互层

**task_agent.py 任务代理**
1. 接收交互层路由下发的任务管理任务与参数
2. 调用任务业务Service完成待办新增、查询、状态变更
3. 任务数据变动时，发布对应事件至事件总线
4. 监听相关联动事件，触发提醒、状态同步等附属逻辑
5. 整理处理结果，向上回传交互界面

### 3.3 Service业务服务层

**person_service.py 人员业务服务**
1. 接收Agent调用，承接人员新增、查询、编辑、删除核心业务
2. 做数据合法性校验、格式规范、业务规则判断
3. 调用对应DAO层方法完成数据存取
4. 封装业务结果组装逻辑，向上返回处理状态与数据
5. **不直接操作数据库底层模型**

**task_service.py 任务业务服务**
1. 接收Agent下发的任务操作指令与参数
2. 校验任务时间、优先级、状态等字段合规性
3. 实现当日任务查询、新增待办、任务状态修改业务逻辑
4. 调用任务DAO完成数据读写
5. 统一封装结果数据，回传给上层代理

### 3.4 DAO数据访问层

**person_dao.py 人员数据访问**
1. 封装人员信息新增、单条查询、列表查询、修改、删除通用方法
2. 承接Service层调用，转换业务数据与数据库存储格式
3. 调用基础模型完成数据库读写
4. 返回原始数据与操作执行状态

**task_dao.py 任务数据访问**
1. 封装任务新增、时间筛选查询、状态更新、任务删除方法
2. 处理查询条件拼装、数据格式适配
3. 对接底层模型完成持久化操作
4. 向上返回数据库原始结果

### 3.5 数据库基础模型层 base_model.py

1. 定义全局统一BaseModel基础父类，包含自增主键、创建时间、更新时间公共字段
2. 编写人员信息数据表结构，匹配既定全部业务字段
3. 编写任务提醒数据表结构，包含所有预设字段
4. 实现数据库初始化、数据表自动创建功能
5. **仅做模型定义与库连接，不处理业务逻辑**

### 3.6 公共核心文件

**event_bus.py 事件总线**
1. 实现事件注册、事件订阅、事件发布、事件监听回调基础能力
2. 维护事件与处理函数映射关系
3. 保证消息收发稳定，不阻塞主线程业务

**main.py 程序入口**
1. 依次初始化数据库模型、事件总线、各层级模块
2. 启动人机交互循环程序
3. 统一捕获全局异常，保障程序平稳运行

**api_server.py HTTP服务入口**
1. 提供FastAPI RESTful API供OpenClaw调用
2. 封装所有业务操作HTTP端点
3. 实现代码读取/写入API（供AI修改代码）
4. 包含架构约束自动检查

## 四、数据库表字段规范

### 4.1 基础公共字段
- 唯一编号(id)、创建时间(created_at)、更新时间(updated_at)

### 4.2 人员表
- 姓名(name)、关系类型(relation_type)、性别(gender)、出生日期(birth_date)、年龄(age)
- 联系电话(phone)、身份证号(id_card)、户籍地址(household_address)
- 工作单位(work_unit)、职务(position)、是否紧急联系人(is_emergency_contact)、备注(remark)

### 4.3 任务表
- 任务标题(title)、任务类型(task_type)、优先级(priority)
- 开始时间(start_time)、截止时间(end_time)、提前提醒时长(remind_before)
- 重复周期(repeat_cycle)、任务状态(status)、任务备注(remark)

## 五、代码编写强制规范

1. 代码兼容本机Python环境，SQLAlchemy语法标准合规，无报错可直接运行
2. 妥善处理中文编码、特殊符号问题，控制台中文显示正常
3. 模块导入路径匹配目录结构，层级调用关系通顺无报错
4. 仅实现既定人员、任务两大业务，不额外扩充无关功能
5. 代码注释、命名风格项目统一，操作后控制台输出中文结果提示
6. **禁止跨层级直接越级调用，严格按照层级依次调用执行**

## 六、运行启动要求

入口文件启动后自动初始化库表与总线，自动进入人机对话交互界面，同步调用、事件监听双机制同时生效，完整响应各类操作指令。

## 七、AI修改代码的强制检查清单

在通过 `/api/v1/code/write` 写入任何代码前，必须检查：

- [ ] 新增文件是否放在正确的层级文件夹中？
- [ ] 是否引入了Python/SQLAlchemy/SQLite以外的技术？
- [ ] 是否存在跨层级直接调用（如Service直接操作数据库）？
- [ ] 是否使用了事件总线替代正常业务调用？
- [ ] 是否包含中文输出反馈？
- [ ] 是否保持了增量新增，未破坏原有代码？
- [ ] 数据库模型变更后是否提醒重启服务？
