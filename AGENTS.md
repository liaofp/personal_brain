# Personal Brain 项目级总约束条件

## 强制生效声明

本文档为 **Personal Brain** 个人大脑事务管理系统的项目级总约束条件。当 OpenClaw Agent 处理与 Personal Brain 相关的任何任务时（包括但不限于代码生成、代码修改、功能扩展、架构调整），必须严格遵守本文档的所有约束。本文档优先级高于任何默认编码习惯。

## 一、基本项目定位

- **项目**: Personal Brain —— 本地Python个人大脑事务管理系统
- **运行环境**: Windows10 / Linux / macOS 本机环境
- **技术栈**: Python + SQLAlchemy + SQLite
- **管控方式**: OpenClaw本地网关调度，调用云端编码模型生成代码
- **核心原则**: 所有开发、迭代、调试、修改均严格遵守本约束，禁止私自变更架构、技术栈、分层逻辑与通信规则

## 二、固定五层架构强制约束

统一从上至下固定层级顺序：

```
人机交互层 → Agent智能代理层 → Service业务服务层 → DAO数据访问层 → 数据库基础模型层
```

### 2.1 层级规则

1. **严禁删减、合并、调换任意层级顺序**，所有代码模块必须归属对应层级，无游离代码文件
2. **区分主动调用与事件联动两种运行模式**，两套机制并行运作、互不冲突
3. **严格分层职责边界**，各层级只处理自身范围内业务，不得越层处理逻辑与数据

### 2.2 各层级职责

| 层级 | 文件夹 | 职责 | 禁止行为 |
|------|--------|------|----------|
| **人机交互层** | `interaction/` | 接收用户输入、意图识别、路由分发、结果展示 | 不编写业务运算、数据存取逻辑 |
| **Agent代理层** | `agent/` | 接收路由、调用Service、发布/订阅事件 | 不直接调用DAO或操作数据库 |
| **Service业务层** | `service/` | 业务逻辑、数据校验、规则判断 | 不直接操作数据库底层模型 |
| **DAO数据层** | `dao/` | 封装CRUD、格式转换 | 不处理业务逻辑 |
| **数据库模型层** | `db_model/` | 模型定义、基类、建表 | 不处理业务逻辑 |

## 三、双模式运行规则与事件总线约束

### 3.1 同步主动业务执行链路

```
人机输入 → 大模型语义意图识别 → 内部路由分发 → 指派对应Agent处理 → 调用Service业务层 → 调用DAO数据访问层 → 读写数据库基础模型层
```

### 3.2 事件总线异步联动链路

```
任意模块数据/业务状态变动 → 主动向事件总线发布事件 → 其他已订阅模块监听 → 自主触发内部Service逻辑
```

### 3.3 总线通用规范

1. 系统统一使用自研 `EventBus` 事件总线，**仅用于状态广播、跨模块联动通知**
2. **禁止用事件总线替代常规业务逐层调用逻辑**
3. 模块按需订阅自身关联事件，不滥用全局监听
4. 新增事件标识、事件触发规则必须确认后再编码
5. 程序启动顺序：**初始化基础模型 → 初始化事件总线 → 加载各层级模块 → 启动人机交互**

## 四、技术栈固定约束

1. 全程仅生成 **Python** 代码，兼容本机运行环境
2. 数据库固定使用：**SQLite** 轻量化数据库
3. ORM框架固定使用：**SQLAlchemy**，严格遵循官方标准语法
4. 数据库层统一封装 `BaseModel` 基础父类，所有业务数据表**强制继承该基类**
5. **禁止替换数据库、ORM框架、编程语言，禁止引入无关第三方库，禁止手写原生SQL**

## 五、强制文件夹与文件结构约束

```
personal_brain/
├── main.py                    # 项目统一启动入口（控制台模式）
├── api_server.py              # FastAPI HTTP服务入口（API模式）
├── event_bus.py               # 全局事件总线核心文件
├── AGENTS.md                  # 项目级总约束条件（本文档）
├── interaction/               # 人机交互层文件夹
│   └── interaction_layer.py
├── agent/                     # Agent智能代理层文件夹
│   ├── person_agent.py
│   └── task_agent.py
├── service/                   # Service业务服务层文件夹
│   ├── person_service.py
│   └── task_service.py
├── dao/                       # DAO数据访问层文件夹
│   ├── person_dao.py
│   └── task_dao.py
├── db_model/                  # 数据库基础模型层文件夹
│   └── base_model.py
└── data/                      # 数据库文件存放目录
    └── brain_data.db
```

**严禁随意新建、删减、移动、合并文件夹与文件。**

新增功能模块时，必须按层级创建对应文件，例如新增「学习计划」功能：
- `agent/study_agent.py` —— Agent层
- `service/study_service.py` —— Service层
- `dao/study_dao.py` —— DAO层
- **不得**在 `db_model/` 外定义模型，**不得**在 `service/` 中直接操作数据库

## 六、迭代与修改红线

1. **已调试正常运行的模块代码，禁止擅自修改、删除、重构**
2. **功能拓展仅使用增量新增方式**，不破坏原有两套运行机制
3. **不得简化分层、删减异常处理、去除日志反馈、缩减交互提示**
4. **所有修改、新增功能，必须先确认需求，再生成代码**
5. 修改代码时，必须保持原有五层架构完整，不得越层调用

## 七、调试与异常处理约束

1. 程序出现报错、无响应、数据存取异常时，自动排查路由分发、主动调用链路、事件监听链路
2. 以中文通俗语言反馈问题原因
3. 代码变更自动留存版本，支持一键回滚稳定版本
4. **所有操作执行后，控制台必须输出中文结果反馈**

## 八、业务边界约束

本项目核心功能仅限：**家人人员信息管理、日常任务提醒管理**

禁止私自拓展额外无关业务功能，仅在确认范围内实现需求逻辑。

**新增功能（如学习计划）必须复用现有 `task` 表结构，通过 `task_type` 字段区分类型，不得新建独立业务表，除非经明确确认。**

## 九、OpenClaw 集成规范

### 9.1 非侵入式集成原则

- **禁止修改 OpenClaw 核心源码**
- **禁止创建 OpenClaw 内部插件**
- 仅通过 **HTTP API** 与 Personal Brain 通信
- Personal Brain 作为**独立进程**运行，OpenClaw 通过 `shell` 工具调用 curl 访问 API

### 9.2 AI 修改 Personal Brain 代码的流程

当用户要求修改 Personal Brain 功能时，AI 必须按以下流程执行：

1. **读取现有代码** —— 通过 HTTP API (`/api/v1/code/read`) 读取相关文件
2. **分析架构影响** —— 确认修改涉及哪些层级，是否违反五层架构
3. **逐层修改代码** —— 按 db_model → dao → service → agent → api_server 顺序修改
4. **重启服务验证** —— 修改后重启 Personal Brain 服务，验证功能正常
5. **中文反馈结果** —— 向用户报告修改内容和验证结果

### 9.3 代码修改安全约束

- 代码写入 API (`/api/v1/code/write`) 限制只能在 `personal_brain/` 目录内
- 禁止目录遍历攻击（路径安全检查）
- 修改数据库模型后，必须重启服务使变更生效

## 十、编码强制规范

1. 代码兼容本机Python环境，SQLAlchemy语法标准合规
2. 妥善处理中文编码、特殊符号问题，控制台中文显示正常
3. 模块导入路径匹配目录结构，层级调用关系通顺
4. 仅实现既定人员、任务两大业务（及经确认的学习计划等扩展），不额外扩充无关功能
5. 代码注释、命名风格项目统一，操作后控制台输出中文结果提示
6. **禁止跨层级直接越级调用**，严格按照层级依次调用执行
